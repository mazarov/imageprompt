# Требования: Google-авторизация без Supabase Auth (GoTrue)

| Поле | Значение |
|------|----------|
| **Статус** | Черновик требований |
| **Дата** | 2026-04-03 |
| **Продукт** | [imageprompt.tools](https://www.imageprompt.tools/) |

## Обзор для читателя

Документ задаёт требования к переходу с **Supabase Auth (GoTrue)** на **собственный** поток Google OAuth и сессии на стороне лендинга (Next.js). База **Postgres/Storage Supabase** может оставаться; меняется только способ входа, хранения профиля приложения и проверки запросов (cookie + Bearer). Ниже — границы, данные, потоки, безопасность и критерии приёмки. Часть решений зафиксирована в блоке **«Принятые решения»** ниже; остаётся **аудит RLS** в прод-БД (§9.1).

### Принятые решения (продукт и упрощённая реализация)

| Тема | Решение |
|------|---------|
| **Сессия (бывш. вопрос 5)** | **Один подписанный JWT приложения** (например HS256), в claims: `sub` = внутренний `user_id`, `iat`, `exp`, опционально `jti` для будущего отзыва. **Отдельный refresh-токен в v1 не вводить** — проще: по истечении срока снова вход через Google. Срок жизни JWT — конфиг (ориентир **7 суток**, подобрать по UX). |
| **Доставка** | **Сайт:** JWT в **httpOnly, Secure, SameSite=Lax** cookie. **Расширение:** тот же JWT в `chrome.storage` + `Authorization: Bearer`. |
| **Logout v1** | Очистка cookie / storage на клиенте; серверная инвалидация по `jti` или таблице сессий — **опционально** в первой итерации (до появления требования «разлогинить везде»). |
| **Клиент расширения и public-config (бывш. вопрос 7)** | Расширение **не** использует Supabase JS для входа и **не** должно зависеть от anon key ради auth. Все защищённые вызовы — **на origin лендинга** с Bearer. **public-config:** убрать поля, нужные только старому Supabase Auth; если прямой доступ к Supabase из расширения не нужен — **не отдавать** `supabaseUrl` / `supabaseAnonKey`. Файлы/Storage — **только signed URL с сервера** через API при необходимости. |
| **Домен www / apex (бывш. вопрос 8)** | Оба варианта допустимы для пользователя: в приложении — **редирект 301** между `https://imageprompt.tools` и `https://www.imageprompt.tools` на **один канонический** host (выбрать один в `NEXT_PUBLIC_SITE_URL`). В **Google Cloud Console** зарегистрировать **оба** Authorized JavaScript origin и **полный набор** redirect URI для канона **и** зеркала (чтобы OAuth не ломался с «не того» host). |
| **Миграция с Supabase Auth (бывш. вопрос 9)** | Пока **нет пользователей** — **отдельную миграцию auth → новые id не делаем**; первый прод сразу на новой таблице и JWT. Карта `auth.users.id` → новый `user_id` понадобится **только если** позже появятся данные, привязанные к старым id. |
| **GDPR / удаление аккаунта (бывш. вопрос 10)** | **Не входит в scope v1**; отдельная задача позже. |
| **Одноразовый exchange после OAuth расширения** | Хранить в **Postgres** (отдельная таблица или поле с TTL), без Redis в v1 — проще инфраструктура. |
| **Таблица пользователей vs `landing_users`** | **Принято вариант A** (§2.1): канонический пользователь в **отдельной таблице**; `landing_users` при необходимости связать FK или постепенно свести кредиты в одну модель. |

### Глоссарий

| Термин | Смысл |
|--------|--------|
| **google_sub** | Стабильный идентификатор аккаунта Google из поля `sub` в id_token / UserInfo; основной ключ связи «Google → пользователь приложения». |
| **PKCE** | Proof Key for Code Exchange; обязателен для публичного OAuth-клиента (расширение) без `client_secret` на клиенте. |
| **Сессия приложения** | Собственные токены/куки приложения (не JWT Supabase Auth), по которым API узнаёт `user_id`. |
| **Service role** | Серверный ключ Supabase с обходом RLS; используется, когда проверка прав перенесена полностью в API. |

## 1. Цель и границы

**Цель:** единая модель «пользователь ImagePrompt» для сайта и Chrome-расширения: Google OAuth, выдача и проверка сессии на приложении, без вызовов Supabase Auth API.

**В scope:** регистрация/вход Google, сессия, профиль в **отдельной таблице** пользователя (§2.1, вариант A), аутентификация API в том же смысле, что сейчас даёт `getSupabaseUserForApiRoute` (cookie на сайте + `Authorization: Bearer` в расширении).

**Вне scope (если не оговорено отдельно):** magic link, пароли, другие IdP; полный отказ от Supabase как БД/Storage — только от **Auth**.

## 3. Текущая связка и миграция идентификатора

Сейчас в коде `user.id` приходит из Supabase Auth и связывается с `landing_users`, `prompt_cards.author_user_id` и др.

**Принято:** пока **нет пользователей в проде**, **отдельную миграцию** с `auth.users` **не выполняем** — запуск с **новыми UUID** в таблице пользователя приложения и JWT `sub`.

**Требование на будущее:** один первичный ключ пользователя на всю модель. Если позже появятся данные, привязанные к старым `auth.users.id`, понадобится **карта соответствия** старый id → новый `user_id` и скрипты переноса (`prompt_cards`, кредиты, биллинг и т.д.).

## 4. Модель данных

### 4.1. Таблица пользователя приложения

Имя таблицы на усмотрение (например `imageprompt_users` / `app_users`); отдельная таблица — **обязательна** (§2.1).

| Требование | Детали |
|------------|--------|
| Уникальность Google | Уникальный индекс по **`google_sub`**. |
| Email | Нормализованный `email`, опционально `email_verified`, `name`, `picture_url`. |
| Временные метки | `created_at`, `updated_at`, `last_login_at`. |
| Статус | При необходимости: `banned`, `deleted_at` (soft delete). |
| Биллинг / кредиты | Поля здесь или FK на `landing_users` по `user_id` (§2.1). |

**Ограничение:** не использовать email как единственный первичный идентификатор; источник истины — **`google_sub`**.

**Миграция учёток с Supabase Auth:** не требуется до появления реальных пользователей (§3); позже — отдельный эпик.

### 4.2. Таблица сессий и одноразовый exchange

**Сессии v1:** отдельная таблица **`sessions` не обязательна**, если достаточно stateless JWT и logout только на клиенте. Ввести таблицу **`sessions`** (или blacklist по `jti`), когда понадобится серверный отзыв (см. «Принятые решения»).

**Одноразовый exchange** после OAuth расширения (§5.2): только **Postgres** (отдельная таблица с `expires_at`, **single-use**), TTL рекомендация **60–120 с**, значение токена **не логировать**.

**Опционально — схема таблицы `sessions`** (когда включён серверный отзыв):

| Поле | Назначение |
|------|------------|
| `id` | Идентификатор сессии (`sid`) |
| `user_id` | FK на пользователя приложения |
| `expires_at` | Срок действия |
| `created_at` | Создание |
| `revoked_at` | Инвалидация (logout, админ) |
| `user_agent_hash` (опц.) | Ограничение/аудит без хранения полного UA |

## 5. Потоки OAuth (без GoTrue)

### 5.1. Сайт (first-party, imageprompt.tools)

- **Authorization Code** для веб-клиента Google; обмен `code` → токены только **на сервере** (`client_secret` не в браузере).
- Успех: upsert пользователя, выпуск сессии приложения (§6), cookie **httpOnly, Secure, SameSite**.

**CSRF для cookie-сессии (обязательно уточнить в реализации):**

- Базово: **`SameSite=Lax`** для cookie сессии (или явная стратегия для cross-site, если появятся сценарии вне first-party).
- Для **state-changing** запросов (POST/PUT/PATCH/DELETE): **double-submit CSRF token**, либо проверка **`Origin` / `Referer`** + допустимые методы; не полагаться только на cookie.
- OAuth **callback** по GET не должен сам по себе создавать долгоживущую сессию без валидации одноразового параметра / серверного обмена кода (как в стандартном code flow).

### 5.2. Chrome-расширение — референс-поток (рекомендация)

Цель: redirect URI в Google Console остаётся на **HTTPS origin** лендинга; расширение не получает `client_secret` и долгоживущий Google refresh.

1. **Старт:** расширение открывает вкладку на страницу лендинга, например `https://www.imageprompt.tools/auth/extension/start` (точный path — в конфиге), с криптостойким **`state`** (и при необходимости привязкой к установке).
2. Дальше стандартный OAuth: **PKCE** (`code_verifier` / `code_challenge`); redirect после Google — на **тот же origin** (зарегистрированный в Google Cloud).
3. **Сервер:** обмен `code` на токены Google, валидация id_token (iss, aud, exp), upsert пользователя, выпуск **одноразового `exchange_token`** (TTL рекомендуется **60–120 с**, single-use; хранение — **Postgres**, §4.2).
4. **Завершение:** страница на лендинге передаёт результат в расширение преимущественно через **`postMessage`** с **allowlist origin** `chrome-extension://<EXTENSION_ID>` (и закрытие вкладки). Редирект на `chrome-extension://.../auth-complete.html?token=...` с долгоживущим JWT в query **не рекомендуется**; если без query никак — только одноразовый код и немедленная замена на сессию внутри расширения.

**Требования:** не логировать одноразовые и долгоживущие токены; не передавать **долгоживущий** JWT приложения в query string.

**Альтернатива (B):** иная схема redirect URI, согласованная с политикой Google OAuth и Chrome — только если референс-поток технически недоступен.

**Запрещено в расширении:** хранить **client_secret** и **refresh token Google**; долгоживущая сессия — только **собственная** (§6).

## 6. Сессия приложения (замена Bearer от Supabase)

Согласовано с блоком **«Принятые решения»**: **один подписанный JWT** на сессию, **без refresh-токена в v1**.

### 6.1. Модель v1

- Подпись **HS256** или **RS256**; claims: `sub` = внутренний `user_id`, `iat`, `exp`, опционально **`jti`** (для будущего отзыва).
- Срок жизни — env (ориентир **7 суток**); по истечении — повторный вход через Google.
- **Refresh-токен не вводить** в первой версии.

### 6.2. Доставка на клиенты

| Клиент | Хранение |
|--------|----------|
| Веб | Значение JWT в **httpOnly, Secure, SameSite=Lax** cookie; `Set-Cookie` + `Cache-Control: no-store`. |
| Расширение | JWT в `chrome.storage` после обмена одноразового кода (§5.2); запросы с `Authorization: Bearer`. |

### 6.3. API

Защищённые Route Handlers: **cookie** (сайт) **или** `Authorization: Bearer <app_jwt>` (расширение) — семантический аналог [landing/src/lib/supabase-route-auth.ts](landing/src/lib/supabase-route-auth.ts).

### 6.4. Logout и эксплуатация (v1)

- **Минимум:** удаление cookie и очистка `chrome.storage` на клиенте.
- **Опционально позже:** серверный отзыв по `jti` или таблица `sessions` (§4.2).
- **Ротация секрета подписи:** процедура с grace period или `kid` в заголовке JWT.
- **Clock skew** при проверке `exp`: рекомендация **±60 с**.

### 6.5. Альтернатива (не v1)

Opaque session id + только таблица `sessions` без JWT — возможна как другая ветка; текущий эпик ведётся по §6.1.

## 7. Безопасность

- **`state`** и при необходимости **`nonce`** (OIDC) на каждый старт OAuth.
- Redirect URI — **строгий allowlist** в коде и в Google Cloud Console (совпадение с §8).
- Валидация **iss, aud, exp** для id_token.
- Rate limiting на `/auth/*` и endpoints обмена кода / одноразового токена.
- Секреты только в env: `GOOGLE_CLIENT_ID`, `GOOGLE_CLIENT_SECRET` (сервер), секрет(ы) подписи JWT.
- CORS: существующий allowlist (`CHROME_EXTENSION_ID` / `CORS_ALLOWED_ORIGINS`).
- **CSP** на страницах auth (минимально необходимая политика; ужесточение по мере готовности).
- Ответы, выдающие сессию или одноразовые токены: заголовки **против кэширования** (`Cache-Control: no-store` и аналоги).

## 8. Google Cloud

- Тип клиента: **Web application** для серверного обмена code → tokens; расширение использует тот же поток через вкладку лендинга (§5.2), без отдельного секрета на клиенте.
- **Authorized JavaScript origins** и **Authorized redirect URIs** для prod, staging, localhost.

Примеры путей (точный список — в env/конфиге и должен **совпадать** с Google Console). Учитывая **и www, и apex**, зарегистрировать **оба** набора (чтобы вход не ломался при заходе с «не канонического» host до редиректа):

- `https://www.imageprompt.tools/...` и `https://imageprompt.tools/...` — те же path для callback (например `/auth/google/callback`, `/auth/extension/callback` или единый callback + `state`)
- Staging: оба варианта host, если применимо
- Локальная разработка: `http://localhost:<PORT>/auth/...`

**Authorized JavaScript origins** — также для **обоих** prod-hostов при необходимости.

## 9. Supabase после отказа от Auth

- Запросы к Postgres/Storage с сервера — через **service role** или выделенный DB-пользователь; **не** опираться на JWT Supabase Auth в RLS для продуктовых проверок.
- **Storage:** выдача **signed URL** и проверка прав на bucket остаются **серверными** (или через политики без `auth.uid()` от старого Auth).

### 9.1. Чеклист аудита RLS

Обязательно: в Supabase Dashboard (или SQL-миграциях) выявить все таблицы с политиками, использующими **`auth.uid()`** или claims Supabase JWT. Для каждой таблицы зафиксировать решение:

- доступ только через API с **service role** + проверка `user_id` в коде, или  
- ослабление/удаление конкретной политики с компенсирующим контролем в API, или  
- **custom JWT** в Postgres (отдельная сложная задача).

Примеры таблиц, которые в коде приложения завязаны на `user.id` и потенциально на RLS (не исчерпывающий список — дополнить по аудиту): `landing_users`, `prompt_cards`, таблицы генераций/медиа по текущим маршрутам API.

## 10. Контракты API (миграция кода)

- Ввести **`getAppUser(request)`** → `{ userId, email?, … } | null` вместо `supabase.auth.getUser()` в защищённых маршрутах.
- Обновить эндпоинты (`/api/me`, `/api/generate`, `/api/vibe/*`, `/api/generations*`, …) под новый `user_id` и схему из §2–4.
- **[landing/src/app/api/public-config/route.ts](landing/src/app/api/public-config/route.ts)** и клиент расширения — **принято для v1:**

  - Расширение **не** использует Supabase JS для auth; защищённые вызовы только на origin лендинга с **Bearer**.
  - Из **public-config** убрать `supabaseUrl` / `supabaseAnonKey`, если они были нужны только для входа через Supabase. Доступ к файлам — **через API** (signed URL с сервера), без anon key в расширении.

## 11. Нефункциональные требования

- Логирование успешных и неуспешных входов **без** токенов и секретов в логах; политика хранения и маскирования PII.
- Мониторинг: рост **401/5xx** на `/auth/*` и пороги алертов (конкретика — по инфраструктуре).
- Построение redirect URL: **канонический** host в **`NEXT_PUBLIC_SITE_URL`** + **301** между www и apex; OAuth redirect URI в Google — для **обоих** hostов (см. §8).
- Опционально: feature flag на период внедрения.
- **GDPR** (удаление аккаунта, экспорт данных): **вне scope v1**, отдельная задача позже.

---

## 12. Критерии приёмки

1. Вход с сайта через Google → сессия в cookie → защищённые API узнают пользователя.
2. Вход из расширения → сессия приложения → `Authorization: Bearer` работает по смыслу как сейчас.
3. Повторный вход того же Google-аккаунта обновляет ту же строку пользователя (upsert по `google_sub`).
4. Нет обращений к **Supabase Auth (GoTrue)** для логина и валидации сессии.
5. Учётные данные продукта хранятся в согласованной с §2–4 схеме (выделенная таблица и/или `landing_users` с FK).
6. В расширении отсутствуют вызовы **`signInWithOAuth` / `exchangeCodeForSession`** у Supabase-клиента.
7. **Logout** очищает cookie расширения/сайта (клиент); серверный отзыв JWT — опционально в v1 (§6.4).
8. В референс-потоке расширения (§5.2) **долгоживущий** токен приложения не передаётся в query string; одноразовый exchange не попадает в логи.
9. Список redirect URI в коде/env **совпадает** с настройками Google Cloud Console для каждого окружения.
